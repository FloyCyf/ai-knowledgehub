## 0. 通用审查框架（所有阶段适用）

为避免每次审查都"拍脑袋"，我先把通用审查维度定下来。每个阶段都会**按这 6 个维度逐项打分**，每项满分 100，总分 = 加权平均。

| 维度 | 权重 | 关注点 | 工具 / 方法 |
| --- | --- | --- | --- |
| **D1. 规范符合度** | 20% | 是否符合 `docs/coding-standards.md` / `docs/api-spec.md` / `docs/database-design.md` | 人工 review + grep |
| **D2. 代码质量** | 20% | 命名、注释、复杂度、重复代码、异常处理 | IDE Inspections + 人工 |
| **D3. 功能完整性** | 25% | 接口是否全实现、边界 case 是否覆盖 | Postman / curl + JUnit |
| **D4. 安全性** | 15% | 鉴权、注入、密码、Token、限流 | OWASP Top 10 自检 + 渗透测试 |
| **D5. 集成与可运行性** | 10% | `mvn install` 成功、能正常启动、跨服务调用通 | 实机 / Docker |
| **D6. 文档与可维护性** | 10% | Swagger 完整、注释到位、README 更新 | Knife4j 验证 + 文档审 |

每个阶段给出**通过/有条件通过/不通过**三档判定：
- **通过**：所有维度 ≥ 80 分
- **有条件通过**：存在 1 项 60~80 分，需在 24h 内修复
- **不通过**：存在 1 项 < 60 分，或阻塞性问题

### 3.3 关键审查点：JWT 解析的 WebFlux 兼容

**易错点**：`common/JwtUtil` 用的是 `Jwts.parser().verifyWith(...).build().parseSignedClaims(token).getPayload()`，**这是同步阻塞调用**。在 WebFlux 的 Gateway 中应该：

```java
// 错误（阻塞 Reactor 线程）
return exchange.getPrincipal().flatMap(p -> parseToken(p));

// 正确（用 Mono.fromCallable + subscribeOn）
return Mono.fromCallable(() -> JwtUtil.parseToken(token))
        .subscribeOn(Schedulers.boundedElastic())
        .flatMap(claims -> chain.filter(exchange));
```

### 4.3 Lua 脚本审查（关键！）

```lua
-- 限流 Lua 脚本
-- KEYS[1] = rate_limit_key
-- ARGV[1] = window_seconds
-- ARGV[2] = max_requests

local current = redis.call('INCR', KEYS[1])
if current == 1 then
    redis.call('EXPIRE', KEYS[1], ARGV[1])
end
return current
```

**审查点**：
- [ ] `INCR` 是原子操作 ✓
- [ ] `EXPIRE` 仅在 first INCR 时设置（避免每次重置 TTL）✓

- [ ] 没有 `KEYS *` 这种 O(N) 操作
- [ ] 没有 `SLEEP` / `WAIT`（避免 Redis 阻塞）

### 7.1 Git 提交规范

```text
feat(gateway): add JWT auth filter
fix(user): fix duplicate username check
docs(readme): update service port table
refactor(common): extract AuthException
test(user): add UserService unit tests
chore(deps): bump spring boot to 3.2.6
```

**审查**：
- [ ] commit message 含 type（feat/fix/docs/refactor/test/chore）
- [ ] commit message 含 scope（gateway/user/article/...）
- [ ] 每次 commit 颗粒度合理（一个 commit 一个功能点）
- [ ] 没有"巨型 commit"（一次提交 100+ 文件）

## 11. 总结

作为 Reviewer，我的原则：

1. **代码规范比个人风格重要** — 100% 严格按 `docs/coding-standards.md`
2. **跑得通的代码 > 漂亮的代码** — 必须能 `mvn install` 一次成功
3. **错误码与文档一致 > 业务逻辑正确** — 老师主要看接口规不规范
4. **注释和 Javadoc > 复杂实现** — 老师要能看懂
5. **生产环境思考 > 当前实现** — 报告是加分项

**重要**：每个阶段的"不通过红线"必须全过，否则不允许进入下一阶段。
